# Sovereign P2P Share

> **Zero-knowledge, direct browser-to-browser encrypted file transfer — packaged as a standalone desktop app.**

No cloud storage. No third-party servers that see your data. Files are encrypted in the browser before transmission and travel directly between peers using WebRTC. The signaling server coordinates the connection but never sees file contents or encryption keys.

## How It Works

### Step-by-step transfer flow

1. **Sender opens the app** → Electron starts an Express/Socket.IO signaling server on `localhost:3001` and an SSH reverse tunnel to `localhost.run`, exposing the server as `https://xxx.lhr.life`.

2. **Sender clicks "Send a File"** → A 6-character cryptographically random Room ID is generated. An AES-GCM-256 key is generated by the browser's `SubtleCrypto` API and embedded in the URL hash: `https://xxx.lhr.life/?room=ABC123#key=<base64url>`.

3. **Sender copies the link** → The app waits for the SSH tunnel to be established (shown as ⏳ tunnel… until ready), then copies the full public URL to the clipboard.

4. **Receiver opens the link** → Their browser connects to the signaling server via the SSH tunnel URL, extracts the key from the URL hash (the hash is **never sent to the server**), and joins the same room as "receiver".

5. **WebRTC handshake** → The server relays SDP offer/answer and ICE candidates between the two peers. Once ICE completes, a direct P2P data channel is established.

6. **File transfer** → The sender encrypts each 64 KB chunk with `AES-GCM-256` and a unique per-chunk IV derived from a session nonce + chunk index. Encrypted frames are sent over the WebRTC data channel directly to the receiver. The signaling server sees none of this.

7. **Receiver saves the file** → Chunks are decrypted and written directly to disk via the File System Access API (`showSaveFilePicker`). An MD5 hash is computed incrementally on the receiver side and compared with the sender's hash for integrity verification.

## Security Model

| Property | Implementation |
|----------|----------------|
| **Zero-knowledge signaling** | The server only relays opaque WebRTC signals. It never sees file chunks or the encryption key. |
| **End-to-end encryption** | AES-GCM-256. The key lives only in the URL hash, which browsers never include in HTTP requests (it is a client-side-only fragment). |
| **Key distribution** | The key is embedded in the share link's `#fragment`. The server never sees the fragment. The key travels from sender to receiver only through the share link itself (out-of-band from the server's perspective). |
| **Integrity verification** | MD5 hash computed incrementally on the sender's original file, sent in the metadata frame, and recomputed on decrypted bytes at the receiver. Mismatch triggers an error. |
| **Tamper detection** | AES-GCM includes a 128-bit authentication tag per chunk. Any in-transit modification causes decryption to throw, aborting the transfer. |
| **Room ID entropy** | 6 characters from a 32-character alphabet using `crypto.getRandomValues()` → ≈30 bits of entropy. Sufficient for session-scoped rendezvous. |
| **Rate limiting** | Express is protected by `express-rate-limit` (100 req/min per IP). Socket.IO connections are rate-limited at 50 messages/sec per socket before disconnect. |
| **Security headers** | `helmet` is applied on all Express routes (CSP disabled to allow the React app's inline scripts; all other headers active). |

### What the signaling server *does* see

- Room IDs (not secret — just rendezvous tokens)
- Socket connection events (logged without socket IDs)
- WebRTC SDP/ICE payloads (contain IP addresses — unavoidable in WebRTC)

### What the signaling server *never* sees

- File contents (transferred directly over WebRTC data channel)
- The AES-256 encryption key (in the URL hash, never sent over HTTP)
- File names or sizes (sent only through the encrypted data channel)
- The session nonce (sent in the encrypted metadata frame)

## Feature Set

### Core Transfer Engine
- ✅ **WebRTC Data Channel** — direct peer-to-peer transfer, no relay for data
- ✅ **64 KB chunked streaming** — constant memory footprint regardless of file size
- ✅ **AES-GCM-256 encryption** — every chunk independently encrypted
- ✅ **Per-chunk IV derivation** — `nonce[8] || chunkIndex[4]` (deterministic, no IV reuse)
- ✅ **MD5 incremental integrity hashing** — rolling hash, verified on receipt
- ✅ **Back-pressure control** — sender waits when WebRTC buffer exceeds 4 MB
- ✅ **Out-of-order chunk buffering** — `DiskWriter` flushes only contiguous runs to disk

### Reliability & Recovery
- ✅ **Bitfield-based resume** — receiver tracks received chunks in a bitfield; resumes from gaps after reconnection
- ✅ **Automatic WebSocket relay fallback** — if P2P ICE fails or times out, the transfer falls back to a WebSocket relay through the signaling server
- ✅ **ICE failure promotion** — stalled `disconnected` state auto-promotes to relay after 20 seconds
- ✅ **Manual reconnect & pause/resume** — sender and receiver can pause mid-transfer and resume

### Storage
- ✅ **File System Access API** (`showSaveFilePicker`) — writes directly to the user's chosen location on disk, no browser sandbox limitation
- ✅ **OPFS fallback** — browsers without File System Access API write to the Origin Private File System then trigger a standard download

### Connectivity
- ✅ **SSH reverse tunnel** (`localhost.run`) — exposes the local signaling server as a public HTTPS URL
- ✅ **STUN servers** — Google STUN servers for NAT traversal
- ✅ **TURN servers** (`openrelay.metered.ca`) — relayed ICE path for symmetric NATs
- ✅ **Tunnel-ready push** — Electron main process pushes the public URL to the renderer via `webContents.send` the instant SSH tunnel establishes (no polling)
- ✅ **Copy-link guard** — "copy link" button is disabled until the tunnel is ready, preventing accidental localhost URL copying

### Electron / Desktop
- ✅ **Self-contained executable** — packaged with `electron-builder`; ships its own Node.js runtime, no user installation of Node required
- ✅ **Linux AppArmor compatibility** — `--no-sandbox`, `--disable-dev-shm-usage`, `--no-zygote` flags prevent Compositor crashes under restrictive AppArmor profiles applied by `.deb` packages
- ✅ **ASAR-safe static serving** — React bundle is unpacked outside the ASAR archive (`asarUnpack`) so `express.static` can serve it using native `fs` calls
- ✅ **Server-ready gate** — `main.js` awaits the Express `listen` callback before creating the `BrowserWindow`, eliminating the race condition blank-screen bug
- ✅ **Key fragment filtered from logs** — renderer console messages containing `key=` are suppressed before forwarding to `journalctl`

## Cryptography Details

### Key Generation
```
key = await crypto.subtle.generateKey({ name: "AES-GCM", length: 256 }, extractable=true, ["encrypt", "decrypt"])
b64key = base64url(await crypto.subtle.exportKey("raw", key))
// Embedded in URL: https://example.com/?room=ABC123#key=<b64key>
```

### IV Derivation (per chunk, deterministic)
```
IV[12] = nonce[0..7] || uint32_big_endian(chunkIndex)[0..3]
```
- The 8-byte `nonce` is randomly generated per session (`crypto.getRandomValues`)
- The 4-byte `chunkIndex` makes each IV unique within the session
- This scheme is safe up to 2³² chunks ≈ 256 TB at 64 KB/chunk

### Wire Frame Format
```
Offset  Size  Field
0       4     chunkIndex    (uint32, big-endian)
4       4     totalChunks   (uint32, big-endian)
8       12    IV            (AES-GCM nonce)
20      N     ciphertext    (AES-GCM encrypted payload + 128-bit auth tag)
```
Total overhead per chunk: **20 bytes header + 16 bytes GCM tag = 36 bytes** on top of 64 KB payload.

## Signaling Server

The server (`server.js`) implements a minimal WebRTC signaling relay:

| Event | Direction | Purpose |
|-------|-----------|---------|
| `join-room` | client→server | Join/create a room as `sender` or `receiver` |
| `peer-joined` | server→client | Notify when the second peer joins |
| `offer` | relay | SDP offer from sender to receiver |
| `answer` | relay | SDP answer from receiver to sender |
| `ice-candidate` | relay | ICE candidates (both directions) |
| `reconnect-signal` | relay | Trigger WebRTC restart |
| `fallback-relay` | server→room | Switch to WebSocket relay mode |
| `relay-data` | relay | Binary chunk relay (fallback mode only) |
| `resume` | relay | Bitfield-based resume coordination |
| `peer-disconnected` | server→client | Peer left the room |

**Room model:** Each room holds exactly two roles: `sender` and `receiver`. A third peer attempting to join receives an error.

## Electron Packaging

### Why `asarUnpack` is required

`electron-builder` packages application files into an `.asar` archive (a virtual read-only filesystem). Node's `express.static()` uses native `fs.readFile` which **cannot read files from inside an `.asar` archive**. The React bundle (`dist/`) must be unpacked to the real filesystem.

```json
"asarUnpack": ["dist/**/*"]
```

The unpacked files land at: `resources/app.asar.unpacked/dist/`

`main.js` resolves the correct path at runtime:
```js
const distPath = app.isPackaged
  ? path.join(process.resourcesPath, "app.asar.unpacked", "dist")
  : path.join(__dirname, "dist");
global.electronDistPath = distPath;  // read by server.js before require()
```

### Linux AppArmor workaround

The `.deb` package installs an AppArmor profile that only grants `userns` capabilities to child processes. Chromium's **zygote** (a pre-forked process spawner) spawns the GPU/Compositor as a sandboxed child that inherits this profile — causing `ESRCH` when it tries to create shared memory in `/dev/shm` or `/tmp`.

Flags applied in `main.js`:
```js
app.commandLine.appendSwitch("no-sandbox");          // Required for packaged Electron on Linux
app.commandLine.appendSwitch("disable-dev-shm-usage"); // Use /tmp instead of /dev/shm
app.commandLine.appendSwitch("no-zygote");           // Skip zygote → no AppArmor profile inheritance
app.disableHardwareAcceleration();                   // Prevent GPU driver blank-screen bugs
```

## Building & Running

### Prerequisites

- Node.js ≥ 18
- npm ≥ 9
- Linux: `fpm` for `.deb` packaging (installed by electron-builder automatically)
- Windows: Run natively or in a Windows CI environment
- macOS: Must be built on macOS for code signing

### Development

```bash
# Install root dependencies
npm install

# Start signaling server + Vite dev server
npm run dev

# Or separately:
npm run server   # Express on localhost:3001
npm run client   # Vite on localhost:5173 (proxies /socket.io → 3001)
```

### Electron (development)

```bash
cd server
npm install
npm run start:electron
```

### Production Build

```bash
# 1. Build the React bundle (outputs to server/dist/)
cd client && npm run build

# 2. Package for Linux (.deb + AppImage)
cd server && npm run package:linux

# 3. Package for Windows (NSIS installer) — run on Windows or with Wine
cd server && npm run package:win

# 4. Package for macOS (DMG) — must run on macOS
cd server && npm run package:mac

# 5. Build Linux + Windows in one command
cd server && npm run package:all
```

Output artifacts land in `server/release/`.

## Platform Notes

### Linux
- Tested on Ubuntu 22.04+ / GNOME
- Installs to `/opt/Sovereign P2P Share/`
- AppArmor profile is applied by the `.deb` post-install script
- Launch from the GNOME application menu or via `sovereign-p2p-share` in terminal
- Logs visible via: `journalctl -xe | grep p2p-share`

### Windows
- NSIS installer with optional install directory
- Creates desktop and Start Menu shortcuts
- No sandbox restrictions; standard Chromium behavior
- Cross-compile from Linux using Wine: `npm run package:win` (experimental)

### macOS
- `.dmg` produced for both Intel (`x64`) and Apple Silicon (`arm64`)
- **Must be built on macOS** — Apple's `codesign` tool is not cross-platform
- Gatekeeper will flag unsigned builds; for distribution, a paid Apple Developer account is required

## Known Limitations

| Limitation | Notes |
|-----------|-------|
| **SSH tunnel dependency** | Requires internet access and `ssh` in `$PATH`. `localhost.run` is a free service with no uptime guarantee. |
| **Tunnel URL is ephemeral** | The `lhr.life` URL changes every time the app restarts. |
| **Single active room** | The signaling server supports multiple concurrent rooms, but the Electron app UI manages one room at a time. |
| **Receiver needs a modern browser** | Requires WebRTC, File System Access API (Chrome 86+, Edge 86+; Firefox/Safari use OPFS fallback). |
| **MD5 for integrity, not authentication** | MD5 is used for file integrity checking only, not for cryptographic authentication (that role is played by AES-GCM's auth tag). |
| **No resume across app restarts** | The bitfield resume mechanism works within a session (reconnections) but not across full app restarts, as the session nonce and keys are ephemeral. |
| **Windows cross-compile** | `npm run package:win` from Linux requires Wine and may miss code-signing. Native Windows build is recommended for distribution. |
