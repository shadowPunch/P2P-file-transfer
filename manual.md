# Sovereign P2P Share: Technical Manual

This document details the internal architecture, cryptographic security model, and feature set of the Sovereign P2P Share application. It covers both the **Standalone Electron Desktop Application** and the **Cloud-Hosted Web Application (Vercel/Railway)**.

---

## 1. Core Feature Set
Regardless of how the application is deployed, the core WebRTC and cryptographic engine provide the following capabilities:

*   **Zero-Knowledge P2P Transfer**: File chunks travel directly between peers over WebRTC. The signaling server coordinates the connection but never sees the file payload.
*   **End-to-End Encryption**: 100% of file chunks are encrypted using **AES-GCM-256** before they leave the browser.
*   **Secure Key Distribution**: The encryption key is randomly generated by the sender and embedded strictly in the URL `#hash` fragment. Because browsers never send the URL hash to web servers, the signaling server mathematically cannot decrypt the transfer.
*   **Memory-Safe Chunking**: Files of any size (even >50GB) are streamed through memory in constant **64 KB chunks**, preventing browser OOM (Out Of Memory) crashes.
*   **Direct-to-Disk Streaming**: Using the **File System Access API**, decrypted chunks are streamed directly to the receiver's hard drive, bypassing browser RAM limits. Browsers without this API gracefully fall back to OPFS (Origin Private File System).
*   **Resumable Transfers**: Receivers maintain a `Uint8Array` bitfield mapping the chunks they have received. Upon reconnecting, this bitfield is sent to the sender, allowing the transfer to resume instantly from gaps without restarting.
*   **Automatic Relay Fallback**: If strict NATs or Firewalls block direct WebRTC P2P (ICE failure), the system automatically promotes the transfer to a secure **WebSocket Relay Mode** through the signaling server.
*   **Continuous Integrity Verification**: An MD5 hash is incrementally calculated on the sender's machine and transmitted. The receiver verifies the decrypted bytes against this hash upon completion.

## 2. Architecture: Standalone Desktop App (Electron)

The desktop application is built to be entirely self-sufficient. It bundles the UI, the WebRTC logic, and the signaling server into a single executable.

### Components
1.  **React Renderer**: The user interface (`client/`). Built with Vite, but configured to output its static bundle to the `server/dist/` folder.
2.  **Express/Socket.io Server**: A local Node.js server (`server/server.js`) that runs on `localhost:3001` in the background. It serves the React bundle and handles Socket.io WebRTC signaling.
3.  **Electron Main Process (`main.js`)**:
    *   Bootstraps the Express server.
    *   Opens a secure **SSH Reverse Tunnel** using `localhost.run`. This exposes the local Express server to the public internet securely (`https://xxx.lhr.life`).
    *   Passes this public URL to the React frontend via IPC, allowing the sender to share a link that receivers anywhere in the world can open.

### Technical Packaging Nuances
*   **ASAR Unpacking**: The React static bundle (`dist/`) is excluded from Electron's encrypted `.asar` archive (`"asarUnpack"`) because Express's `express.static()` relies on native OS file reading, which cannot penetrate an ASAR file.
*   **Linux AppArmor Sandbox**: To prevent `ESRCH` crashes on modern Linux distributions (like Ubuntu 24.04), the Electron main process disables the Chromium Zygote process (`--no-zygote`) and uses `/tmp` instead of `/dev/shm` to avoid strict AppArmor restrictions inherited during `.deb` installation.

## 3. Architecture: Cloud-Hosted Web App (Vercel + Railway)

To remove the dependency on a local SSH tunnel and allow users to run the platform purely in the cloud, the codebase was decoupled into a traditional Client/Server split, located in the `hosted version/` directory.

### Frontend (Vercel)
*   **Location**: `hosted version/client`
*   **Changes**: Stripped of all Electron IPC logic and tunnel-waiting screens. 
*   **Configuration**: Uses `import.meta.env.VITE_SERVER_URL` to determine where to send Socket.io traffic. Vercel automatically maps your custom domain, meaning share links are generated instantly using `window.location.href`.

### Backend (Railway / Render)
*   **Location**: `hosted version/server`
*   **Changes**: Stripped of Electron shell wrappers, tunneling dependencies, and static file serving. It acts *purely* as an API and Signaling Relay.
*   **Configuration**: Binds to the cloud provider's internal `PORT`. Uses the `CORS_ORIGIN` environment variable to ensure it only accepts signaling requests from the approved Vercel frontend.

## 4. Cryptography & Wire Protocol

The engine uses Web Crypto API (`crypto.subtle`).

### Key & Nonce Generation
*   **Key**: AES-GCM, length 256, extractable for URL embedding.
*   **Nonce**: An 8-byte random initialization vector generated once per session.

### Per-Chunk Deterministic IV
AES-GCM requires a unique IV for every single encryption operation. The engine deterministically generates a 12-byte IV for each chunk to ensure uniqueness while avoiding sending full IVs over the wire:
`IV[12] = nonce[0..7] || uint32_big_endian(chunkIndex)[0..3]`

### Wire Frame Format
Data channel frames include exactly 36 bytes of overhead per 64 KB chunk.
*   `Bytes 0-3`: Chunk Index (Uint32)
*   `Bytes 4-7`: Total Chunks (Uint32)
*   `Bytes 8-19`: The 12-byte computed IV.
*   `Bytes 20-N`: AES-GCM Ciphertext (which inherently includes a 16-byte authentication tag at the end).

If a chunk is tampered with in transit, the AES-GCM decryption throws an error automatically, guaranteeing absolute data integrity.

## 5. Deployment and Build Instructions

### Building the Desktop Executables
Run the following from the root `p2p-share-phase1-3` directory.
1.  **Compile Frontend**: `cd client && npm install && npm run build`
2.  **Package Linux (`.deb` / `.AppImage`)**: `cd ../server && npm install && npm run package:linux`
3.  **Package Windows (`.exe`)**: `npm run package:win`
4.  **Package macOS (`.zip`)**: `npm run package:mac`
    * *Note: macOS packaging falls back to `.zip` when built on Linux. To generate a native `.dmg`, the build command must be run on a physical Mac.*

### Deploying the Hosted Version
1.  **Railway (Backend)**
    *   Deploy `hosted version/server` to Railway.
    *   Set the `CORS_ORIGIN` environment variable to your future Vercel URL.
2.  **Vercel (Frontend)**
    *   Deploy `hosted version/client` to Vercel.
    *   Set the `VITE_SERVER_URL` environment variable to your live Railway URL.
    *   Redeploy to bake the environment variable into the React bundle.
